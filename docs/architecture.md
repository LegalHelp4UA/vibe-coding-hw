# Архітектура продукту: з яких блоків він складається

Домашнє завдання уроку 5 курсу Vibe Coding. Мета — навчитися пояснювати
будь-який IT-продукт як **систему блоків**, а не як набір випадкових файлів.
Коли розумієш блоки, можна давати AI точні задачі й перевіряти, який саме
блок системи змінився.

## П'ять блоків будь-якого продукту

> **Frontend показує. Backend думає. База пам'ятає. API з'єднує. AI додає інтелект.**

- **Frontend** — те, що бачить користувач: інтерфейс, кнопки, поля. Через нього
  людина взаємодіє з продуктом. Це «вітрина».
- **Backend** — де живе логіка. Приймає запит, перевіряє дані, виконує правила,
  звертається до бази, повертає відповідь. Проста ознака бекенду: там є «якщо».
- **Database (база даних)** — пам'ять продукту. Без бази продукт пам'ятає лише
  поточну дію; з базою має історію: користувачів, операції, налаштування.
- **API** — правила, за якими частини системи спілкуються між собою. Не магія,
  а зрозумілий обмін: `запит → обробка → відповідь`. Як вікно видачі: називаєш
  замовлення за формою й отримуєш результат, не заходячи на кухню.
- **AI-модель** — окремий блок, до якого backend звертається через API, щоб
  проаналізувати, пояснити чи згенерувати. AI сам нічого не пам'ятає — усе, що
  йому треба, передають на вхід.

## На прикладі нашого Telegram-бота

Навчальний проєкт цього репозиторію — інформаційний Telegram-бот на **aiogram**
(команди `/start`, `/help`, `/services`, `/process`, `/contacts`). Навіть такий
простий бот — це вже маленький IT-продукт з архітектурою:

| Блок | Що це в нашому боті |
|------|---------------------|
| **Frontend** | **Telegram** — інтерфейс, який ми не писали, а використовуємо готовим. Користувач бачить чат і кнопки команд. |
| **API** | **Telegram API** — канал між Telegram і нашим кодом. Ми не знаємо, що в Telegram «під капотом», і не мусимо — знаємо лише правила обміну. |
| **Backend** | **Python-застосунок на aiogram** (`app/`). Приймає команду, вирішує, що відповісти, повертає текст. Тут живе вся логіка бота. |
| **Database** | **Поки немає.** Бот відповідає статичним текстом і нічого не пам'ятає між повідомленнями. Коли додамо базу — зможемо зберігати користувачів та історію. |
| **AI-модель** | **Поки не підключено.** Наступний крок розвитку — щоб бот міг не лише віддавати готові тексти, а й відповідати на вільні питання. |

Маршрут одного повідомлення:

```
Користувач пише /services у Telegram
   ↓  Telegram (frontend)
   ↓  Telegram API
Python/aiogram (backend) — обробник команди вирішує, що відповісти
   ↓  Telegram API
Користувач бачить відповідь у Telegram
```

Одна проста дія — і вже видно кілька блоків системи.

## Головна функція, яку ми проєктуємо: помічник для обробки документів

Найкорисніше — розкласти по блоках не абстрактний, а **реальний інструмент, який
ми хочемо спроєктувати для своєї роботи**. Наш продукт — юридичний сервіс, і
велика частина роботи юриста починається з того, що клієнт надсилає документи
(фото паспорта, PDF, скани), з яких треба **витягти дані** й акуратно вписати їх
у документ, який готує юрист.

Ми проєктуємо інструмента-помічника, який робить це замість ручної роботи. У нього
**дві функції**:

- **Функція A — витяг (extract):** з різних завантажених документів дістати
  потрібні дані у структурованому вигляді (наприклад, ПІБ, дані документа,
  реквізити об'єкта, про який ідеться).
- **Функція B — перевірка:** після того як юрист склав проєкт документа —
  звірити, чи правильно витягнуті й вписані дані, а також граматику, стилістику
  й правильність перекладу.

Розкладемо цей інструмент по блоках:

| Блок | Що це в нашому помічнику |
|------|--------------------------|
| **Frontend** | **Telegram** — юрист працює через зручний чат: надсилає файл, отримує результат. Свій інтерфейс ми не пишемо, використовуємо готовий Telegram. |
| **API** | **Три канали, не один:** (1) **Telegram API** — між Telegram і нашим backend; (2) API до **зорової моделі** — щоб «прочитати» зображення й PDF; (3) API до **мовної моделі** — щоб розкласти прочитане у поля й перевірити текст. |
| **Backend** | **Внутрішній сервіс-оркестратор:** приймає файл, по черзі викликає зорову модель, потім мовну, застосовує наші правила (які дані потрібні, як їх перевіряти), збирає відповідь і повертає юристу. Це серце інструмента — логіка, специфічна саме для наших документів. |
| **Database** | **Структурована база (PostgreSQL):** зберігає витягнуті поля, прив'язку до конкретної справи й клієнта, результати перевірки, статус. Без неї дані нікуди не «лягають» і губляться у файлах — тому база тут ключова. |
| **AI-модель** | **Дві різні здатності:** (1) **зорова** (OCR / vision) — прочитати текст із зображення чи PDF; (2) **мовна** (LLM) — структурувати прочитане у потрібні поля й перевірити граматику, стилістику, переклад. Одна модель не робить обидва — це два окремі виклики. |

Маршрут даних (функція A — витяг):

```
Юрист надсилає фото документа в Telegram
   ↓  Telegram (frontend)
   ↓  Telegram API
Backend-сервіс — прийняв файл, запустив обробку
   ↓  API до зорової моделі
AI-зір (OCR) — прочитав текст із зображення
   ↓  API до мовної моделі
AI-мова (LLM) — розклав прочитане у структуровані поля
   ↓
База даних — зберегла поля + прив'язку до справи
   ↓
Backend — повернув юристу структуровані дані на перевірку
```

Функція B (перевірка) йде тим самим маршрутом, тільки на вході — готовий проєкт
документа, а мовна модель звіряє правильність даних, граматику, стиль і переклад.

Що дає такий розбір: одразу видно, що **головне, чого бракує, — це блок пам'яті
(структурована база).** Інструмент може прочитати й розкласти дані, але без бази
їм немає куди системно лягти. Це і є перше архітектурне рішення, яке треба
ухвалити, перш ніж будувати інструмент.

## Ще функції для порівняння

Щоб закріпити мислення, коротко розкладемо звичні функції з популярних застосунків.

### Банківський застосунок — переказ грошей

| Блок | Що відбувається |
|------|-----------------|
| Frontend | Екран переказу: сума, отримувач, кнопка «Надіслати» |
| API | Застосунок передає запит на переказ у банківський backend |
| Backend | Перевіряє: чи є кошти, чи коректний отримувач, чи не перевищено ліміт |
| Database | Списує й зараховує суму, зберігає операцію в історії |
| AI | Перевірка на шахрайство: чи не виглядає переказ підозрілим |

### Netflix — «продовжити перегляд»

| Блок | Що відбувається |
|------|-----------------|
| Frontend | Рядок «Продовжити перегляд» із потрібним моментом |
| API | Застосунок запитує в backend, що і де користувач дивився |
| Backend | Формує список за історією саме цього профілю |
| Database | Зберігає, на якій хвилині зупинився кожен фільм для кожного користувача |
| AI | Рекомендації на основі історії переглядів |

### Таксі — виклик авто

| Блок | Що відбувається |
|------|-----------------|
| Frontend | Карта, точки «звідки/куди», кнопка «Замовити» |
| API | Пошук водія + запити до картографічного сервісу (зовнішній API) |
| Backend | Шукає найближчих водіїв, рахує вартість і час подачі |
| Database | Зберігає поїздку, водіїв, історію замовлень, оплати |
| AI | Прогноз ціни в години пік і підбір водія |

## Висновок

Найважливіша навичка з цього уроку — не запам'ятати слова, а вміти **дати кожній
функції адресу**: який блок показує, який рахує, що зберігається, через який API
йде обмін і де потрібен AI. Тоді замість «зроби мені помічника для документів»
можна поставити задачу точно — «додай backend-сервіс, який приймає файл, викликає
зорову модель для читання, мовну — для структурування у поля, і зберігає їх у базу
з прив'язкою до справи». AI не мусить вгадувати, а ти можеш перевірити, який блок
системи змінився.
