## 2. Docker — запуск проєкту в контейнері

**Docker** запускає програми в ізольованих контейнерах. На Windows працює
поверх **WSL2** (Windows Subsystem for Linux). Спершу довелося доустановити WSL2,
бо Docker Desktop повідомив, що його немає.

```powershell
wsl --install          # встановлення WSL2 (в адмін-PowerShell)
wsl --status           # стан і версія WSL за замовчуванням
wsl --version          # детальна версія движка WSL

docker --version       # перевірка, що Docker CLI встановлено
docker info            # стан движка (поява секції Server: = движок працює)
docker run hello-world # тест: завантажити й запустити крихітний контейнер
```

**Результат:** встановлено WSL 2.7.8 (версія за замовчуванням — 2). Docker
Desktop підхопив WSL, движок піднявся (`Engine running`). `docker run hello-world`
вивів `Hello from Docker!` — увесь ланцюжок працює.

## 3. Nginx-контейнер і перевірка localhost

Запустили справжній веб-сервер **Nginx** у контейнері й відкрили його в браузері.

```powershell
docker run -d -p 8080:80 nginx   # Nginx у фоні, проброс порту 80 -> 8080
docker ps                        # список запущених контейнерів (статус Up, порти)
```

- `-d` — detached, контейнер працює у фоні;
- `-p 8080:80` — порт `80` усередині контейнера показано як `8080` на компʼютері.

**Результат:** `docker ps` показав контейнер `nginx`, статус `Up`,
порти `0.0.0.0:8080->80/tcp`. У браузері за адресою **http://localhost:8080**
відкрилася сторінка «Welcome to nginx!».

Корисні команди для прибирання після перевірки:

```powershell
docker stop <container_id>   # зупинити контейнер (id видно у docker ps)
docker rm <container_id>     # видалити зупинений контейнер
```

## 4. Публічний тунель через ngrok

**ngrok** створює тимчасове публічне посилання на `localhost` — щоб локальний
сайт можна було відкрити з інтернету.

```powershell
winget install ngrok.ngrok             # встановлення ngrok (після — перезапуск VS Code для оновлення PATH)
ngrok version                          # перевірка версії
ngrok update                           # оновлення ngrok (акаунт вимагає актуальну версію)
ngrok config add-authtoken <ВАШ_ТОКЕН> # привʼязка до акаунта (токен — з dashboard.ngrok.com)
ngrok http 8080                        # підняти публічний тунель на порт 8080
```

**Результат:** перша версія з winget (3.3.1) виявилася застарою — акаунт вимагав
мінімум 3.20.0, тож виконали `ngrok update` (оновилось до 3.39.8). Після
`ngrok http 8080` тунель став `online`, у рядку **Forwarding** зʼявився
публічний URL виду `https://<...>.ngrok-free.dev -> http://localhost:8080`.
Відкриття цього URL у браузері (зокрема з мобільної мережі телефона) показало ту
саму сторінку «Welcome to nginx!», а в табличці ngrok було видно живі
HTTP-запити. Зупинка тунелю — `Ctrl+C`.

> ℹ️ Тунель і його URL — тимчасові: посилання живе лише поки запущено
> `ngrok http 8080`. Після зупинки URL стає offline (`ERR_NGROK_3200`) — це
> нормально. Перевірка ДЗ відбувається за посиланням на GitHub, а не за живим
> тунелем.

> 🔒 `authtoken` і конкретний публічний URL — особисті/тимчасові, у репозиторій
> навмисно не вписані. Публічний URL дає прямий доступ до компʼютера на час
> сесії, тому тримали його недовго.

## 5. Cloudflare Tunnel — другий тип тунелю (додаткове завдання)

Додатково спробували **другий тип тунелю** — Cloudflare Tunnel (`cloudflared`),
у режимі **TryCloudflare** (швидкий тунель без реєстрації).

```powershell
winget install Cloudflare.cloudflared              # встановлення cloudflared
cloudflared --version                              # перевірка версії
cloudflared tunnel --url http://localhost:8080     # тимчасовий публічний тунель (без акаунта)
```

**Результат:** `cloudflared` 2026.5.2 встановлено. Команда одразу (без токена)
підняла тунель і видала публічний URL виду `https://<...>.trycloudflare.com`.
Перевірка показала ту саму сторінку Nginx — `HTTP 200` і `<title>Welcome to
nginx!</title>`. Зупинка тунелю — `Ctrl+C`.

### Коротке порівняння: ngrok vs Cloudflare Tunnel

| Критерій | **ngrok** | **Cloudflare Tunnel** (quick) |
|---|---|---|
| Реєстрація / токен | потрібні акаунт + `authtoken` | **не потрібні** (TryCloudflare) |
| Команда запуску | `ngrok http 8080` | `cloudflared tunnel --url http://localhost:8080` |
| Публічний домен | `*.ngrok-free.dev` | `*.trycloudflare.com` |
| Веб-інспектор запитів | є (`http://127.0.0.1:4040`) | немає (лише metrics-endpoint) |
| Запуск «з коробки» | вимагав оновлення агента (`ngrok update`) | запрацював одразу |
| Стабільність/гарантії | вища (акаунт) | без гарантій uptime (для дослідів) |

**Висновок:** для швидкої разової демонстрації зручніший **Cloudflare Tunnel**
(не треба реєстрації й токена). **ngrok** дає корисний веб-інспектор трафіку
(`:4040`) і більше можливостей під акаунтом. Обидва успішно дали публічний доступ
до локального Nginx.

## 6. Git-цикл (вайбкодинг)

Усі експерименти та AI-згенеровані зміни — в окремій гілці, не в `main`.
Цей файл згенеровано саме в межах такого циклу.

```powershell
git checkout -b environment/setup              # створити нову гілку й перейти на неї
git status                                     # подивитися, що змінилося
git add environment-check.md                   # додати файл до коміту
git commit -m "docs: add environment-check.md" # зберегти зміну з осмисленим описом
git push -u origin environment/setup           # відправити гілку на GitHub
# далі — Pull Request на GitHub -> review -> merge у main -> видалити гілку
git checkout main                              # повернутися на main
git pull                                       # підтягнути злиті зміни
```

> ℹ️ Назва гілки — `environment/setup` (як у завданні). Спроба з пробілом
> (`Environment Setup`) дала помилку: git не дозволяє пробіли в назвах гілок,
> а от слеш `/` — дозволений.
